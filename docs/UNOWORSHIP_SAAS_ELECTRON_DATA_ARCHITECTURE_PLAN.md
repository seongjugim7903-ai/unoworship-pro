# UnoWorship SaaS / Electron 데이터 구조 기획

작성일: 2026-07-22

## 1. 목적

UnoWorship은 다음 두 제품 영역을 하나의 서비스처럼 연결하되, 실행 위치와 데이터 책임을 명확히 분리한다.

1. 구독 교회가 웹에서 사용하는 입력·협업 서비스
2. 방송실 컴퓨터에 설치되는 Electron 현장 송출 프로그램

웹 입력 자료, 교회별 클라우드 자료, 현장 프로그램 파일, 성경·찬송가 원본을 같은 저장소로 취급하지 않는다.

### 운영 경계 한눈에 보기

| 영역 | 실행 위치 | 저장 대상 | 저장 금지 대상 |
|---|---|---|---|
| 공개 웹 | Vercel | 제품 소개, 가격, 공개 문서 | 교회 예배 자료, 인증 비밀값 |
| 교회 입력·워크스페이스 | Vercel + Supabase | 교회별 요청, 가사, 악보, 생성 이미지, 프로그램 payload | 다른 교회 자료, 로컬 장치 설정 |
| Electron 현장 앱 | 방송실 컴퓨터 | 가져온 프로그램, 장치 설정, 오프라인 사본, 녹화 | `service_role` 키, 공용 저작권 원본 |
| GitHub | 소스 저장소 | 코드, 테스트, 마이그레이션, 예제 설정, 문서 | 실데이터, 비밀키, 빌드 산출물, 교회 콘텐츠 |

데이터 위치를 결정할 때는 "어느 화면에서 만들었는가"보다 "누가 소유하며 오프라인에서 필요한가"를 기준으로 한다.

## 2. 제품 영역 구분

### A. 공개 웹

- 위치: `unoworship.kr`
- 대상: 방문자, 가입 전 사용자
- 역할: 회사·제품·프라이싱·리소스, 회원가입, 구독 신청
- 교회 예배 자료를 저장하지 않는다.

### B. 구독 교회 워크스페이스

- 위치 예시: `app.unoworship.kr/@ulju`
- 대상: 구독이 활성화된 교회의 승인된 사용자
- 역할: 자막 협조, 대시보드, 캔버스, 설정, 자료 검색 및 현장 전송
- 모든 데이터는 `church_id` 또는 `workspace_id`로 교회별 분리한다.

### C. 교회별 입력페이지 서비스

입력페이지는 공개 도구가 아니라 구독 교회 워크스페이스에 포함되는 서비스다.

초기 입력 유형:

- 헵시바 선교단: 찬양대 가사, 곡 정보, 자막 이미지, 프로그램 payload
- 설교대지: 설교 제목, 본문, 대지·소지, 인용문 및 표시 순서
- 준비찬양: 곡 정보, 가사, 악보, 콘티 및 생성 결과

공통 조건:

- 로그인 사용자가 소속된 교회만 작성·조회·수정할 수 있다.
- 구독이 활성, 체험 또는 허용된 유예 상태일 때만 사용한다.
- URL에 교회명을 적는 것만으로 권한을 판단하지 않는다.
- 생성된 이미지·악보·프로그램 payload도 원본 요청과 같은 교회 소유로 저장한다.
- 교회 관리자, 작성자, 방송실 담당자 등 역할별 권한을 둔다.

### D. Electron 현장 프로그램

- 설치 위치: 교회 방송실 macOS 컴퓨터, 이후 Windows 지원
- 역할: Composer, ATEM Fill/Key, Sub, 카메라 릴레이, 녹화·라이브 제어
- 클라우드에서 교회에 허용된 자료 목록을 읽고 사용자가 선택한 자료만 로컬 프로그램으로 가져온다.
- 인터넷이 끊겨도 이미 내려받은 예배 자료는 계속 송출할 수 있어야 한다.

## 2-1. GitHub 저장소 구성

기존 `unoworship-pro` 저장소는 한 저장소 안에서 두 실행 제품을 분리한다.

```text
unoworship-pro/
  app/                    현재 Vercel 입력 웹앱
  lib/
  supabase/
  docs/
  apps/
    atem-field/           Electron/Next 현장 자막 프로그램
```

웹앱 루트는 현재 Vercel 연결을 유지하기 위해 이동하지 않는다. Electron 소스는 `apps/atem-field`에 독립된 `package.json`, 빌드 설정, 테스트를 가진다. 루트 웹 빌드가 Electron 하위 프로젝트를 타입 검사하거나 배포하지 않도록 `tsconfig`, ESLint, Vercel 제외 규칙을 둔다.

### GitHub에 포함

- TypeScript/React/Next.js/Electron 소스
- `package.json`, lockfile, 빌드 설정
- Supabase 마이그레이션과 RLS 정책
- 테스트와 운영 점검 스크립트
- `.env.example`과 하드웨어 프로필 예제
- 저작권 문제가 없는 기본 UI 템플릿
- 운영·설치·복구 문서

### GitHub에서 제외

- `.env.local`, 인증 토큰, Supabase `service_role` 키
- `node_modules`, `.next`, `dist`, DMG, 로그와 캐시
- 성경·찬송가·CCM 원문 데이터
- 교회 PPT, 악보, 이미지, 영상, 녹화 파일
- `data/programs`의 실제 예배 프로그램
- 생성된 PPT 이미지와 분석·변환 중간 파일
- 특정 교회의 EDID/IP/ATEM 설정 실파일

설치파일은 Git 소스에 커밋하지 않고 버전 태그에 연결된 GitHub Release 자산으로 배포한다.

## 3. 클라우드 멀티테넌트 기준

### 필수 핵심 엔터티

- `churches`: 교회/기관 워크스페이스
- `church_members`: 사용자와 교회, 역할 연결
- `subscriptions`: 플랜, 체험, 결제 및 사용 가능 상태
- `content_requests`: 입력페이지 공통 요청 메타데이터
- 유형별 상세 테이블: 찬양대, 설교대지, 준비찬양
- `generated_assets`: 생성 이미지·악보·PDF 등 Storage 파일 메타데이터
- `program_payloads`: Electron Composer가 가져올 프로그램 구조
- `device_registrations`: 승인된 현장 컴퓨터와 디바이스 토큰
- `import_receipts`: 어떤 장치가 어떤 자료를 언제 가져왔는지 기록

모든 교회 소유 테이블은 최소한 다음 컬럼을 가진다.

```text
church_id
created_by
created_at
updated_at
status
```

Storage 경로도 교회별로 분리한다.

```text
churches/{church_id}/choir/{request_id}/...
churches/{church_id}/sermons/{request_id}/...
churches/{church_id}/praise/{request_id}/...
```

### 접근 통제

- Supabase Auth 사용자와 `church_members`를 연결한다.
- RLS가 로그인 사용자의 `church_id` 소속과 역할을 확인한다.
- `service_role` 키는 Vercel 서버에서만 사용하며 Electron·브라우저·GitHub에 넣지 않는다.
- Electron은 사용자 세션 또는 제한된 디바이스 토큰으로 교회 범위 API만 호출한다.
- 구독 상태는 서버에서 검증하고 클라이언트가 보낸 플랜 값을 신뢰하지 않는다.

## 4. 현재 헵시바 데이터의 위치

현재 구현된 테이블은 다음과 같다.

- `choir_requests`
- `choir_generated_images`
- `choir_programs`
- `choir-generated-images` Storage bucket

현재 스키마에는 교회 구분 컬럼이 없으므로 단일 교회 시험 운영 구조다. 정식 구독 서비스 전에 다음 이관이 필요하다.

1. `church_id`, `created_by` 컬럼 추가
2. 기존 행을 울주교회 또는 실제 소유 교회로 명시적으로 배정
3. Storage 파일을 교회별 경로로 재배치하거나 경로 매핑
4. 교회별 RLS 정책 적용
5. API의 목록·검색·수정·생성 쿼리에 교회 범위 강제

기존 데이터는 삭제하지 않고 이관한다. 소유 교회를 확정하지 못한 데이터는 관리자 검수 대기 상태로 둔다.

## 5. 로컬 데이터와 클라우드 데이터

### 클라우드에 저장

- 입력페이지에서 작성한 원문과 메타데이터
- 교회가 업로드한 악보와 생성 이미지
- 프로그램 생성용 payload
- 작성·수정·가져오기 이력
- 교회 설정 및 권한 정보

### 교회 컴퓨터에만 저장

- 교회가 적법하게 보유한 성경·찬송가 원본 데이터
- 로컬 PPT/Keynote 원본과 변환 이미지
- 대용량 현장 미디어 및 녹화 파일
- 가져온 예배 프로그램의 오프라인 사본
- 출력 장치, ATEM 입력, EDID 등 현장 하드웨어 설정

준비찬양 악보처럼 입력페이지에서 교회가 업로드한 자료는 해당 교회의 클라우드 원본으로 저장할 수 있다. Electron은 그중 현장 사용을 승인한 자료만 로컬 오프라인 사본으로 내려받는다. 반면 성경·찬송가 전체 원본 데이터는 일반 입력자료가 아니며 공용 Supabase, GitHub, 공용 설치파일에 넣지 않는다.

성경·찬송가 원본과 교회별 PPT·미디어는 GitHub와 공용 설치파일에 포함하지 않는다.

권장 로컬 경로:

```text
~/Documents/UnoWorship Library/
  Bibles/
  Hymns/
  Programs/
  PPT/
  Media/
  Recordings/
  Backups/
```

인증 토큰·장치 설정·캐시는 운영체제의 앱 데이터 경로에 둔다.

```text
~/Library/Application Support/UnoWorship/
```

설치된 앱의 `Contents/Resources`, `app.asar`, 프로젝트 내부 `data/`는 읽기 전용으로 간주한다. 프로그램 저장 API와 파일 생성기는 반드시 공통 `LocalLibraryPath` 모듈을 통해 외부 라이브러리 경로를 사용해야 한다.

### 로컬 파일 관리 기준

- 프로그램과 에셋은 임시 파일에 먼저 쓴 뒤 검증 후 원자적으로 교체한다.
- 파일명 충돌은 덮어쓰지 않고 ID 또는 버전으로 관리한다.
- 각 프로그램은 `schemaVersion`, `churchId`, `programId`, `revision`, 에셋 checksum을 가진다.
- 가져오기 중 실패하면 기존 프로그램을 그대로 유지한다.
- 앱 업데이트 전에 로컬 데이터 스키마 마이그레이션과 자동 백업을 수행한다.
- 사용자가 선택한 위치로 라이브러리 전체 백업·복원할 수 있어야 한다.
- 녹화 파일은 프로그램 데이터와 분리하고 용량 부족 경고를 제공한다.

## 6. Electron 가져오기 흐름

1. Electron 앱이 로그인 계정과 디바이스 토큰을 확인한다.
2. 서버가 소속 교회와 구독 상태를 검증한다.
3. Electron은 해당 교회의 `program_payloads` 목록만 조회한다.
4. 방송 담당자가 자료를 선택하고 `가져오기`를 누른다.
5. 필요한 이미지에 짧은 만료시간의 signed URL을 발급한다.
6. 프로그램 JSON과 이미지를 로컬 라이브러리에 원자적으로 저장한다.
7. 저장 검증이 끝난 뒤 Composer 목록에 추가한다.
8. `import_receipts`에 장치와 가져오기 결과를 기록한다.

클라우드 자료가 있다고 자동으로 현재 예배 프로그램에 삽입하지 않는다. 방송 담당자의 명시적 가져오기와 검수 단계를 유지한다.

### 동기화와 충돌 기준

- 클라우드의 `program_payload`를 원본 준비자료로, 로컬로 가져온 파일을 현장 실행 사본으로 본다.
- 같은 `programId + revision`을 다시 가져오면 중복 생성하지 않는다.
- 클라우드에 새 revision이 있으면 `업데이트 있음`으로 표시하고 자동 덮어쓰지 않는다.
- 로컬에서 수정한 프로그램은 사용자가 명시적으로 업로드하기 전까지 클라우드 원본을 변경하지 않는다.
- 삭제는 즉시 물리 삭제하지 않고 휴지통/보존기간을 거친다.
- 교회 변경 또는 로그아웃 시 다른 교회의 클라우드 목록과 캐시를 표시하지 않는다.

## 7. Electron 설치 및 업데이트 원칙

- 1차 배포는 Apple Silicon macOS DMG를 우선한다.
- 앱 코드는 설치파일에 넣고 교회 콘텐츠는 외부 라이브러리에 둔다.
- 업데이트는 초기에는 사용자가 실행하는 수동 업데이트로 운영한다.
- 앱을 교체해도 로컬 라이브러리와 인증 상태가 유지되어야 한다.
- 기존 Chrome 4화면 실행 스크립트는 첫 안정화 기간 동안 비상 복구 수단으로 보존한다.
- 패키지 내부 서버는 `npm`/`tsx` 개발 실행이 아니라 Next.js standalone 런타임으로 기동한다.
- Electron이 Composer, Fill, Key, Sub, 카메라 릴레이 창을 직접 배치하고 관리한다.

현재 Electron 골격은 설치 배포 완료 상태가 아니다. 다음 보완을 선행한다.

- `next.config.ts`에 standalone 출력 적용
- 패키지 안에서 `npm run start`를 호출하지 않고 번들된 Node 서버를 직접 기동
- `electron-builder.files`의 포괄적 `data/**/*` 포함 제거
- `asar` 내부에 쓰려는 모든 코드 제거
- 현재 ATEM 구조에 맞춰 Composer, Fill, Key, Sub, Relay 창 구성
- EDID 기반 디스플레이 매핑 헬퍼를 앱 리소스로 포함하고 교회별 프로필로 분리
- `backgroundThrottling: false`와 전원 차단 방지를 적용하되 카메라 릴레이 부하 측정
- 내부 서버 health check, 포트 충돌 안내, 비정상 종료 복구
- macOS 카메라·마이크 권한, 코드 서명, 공증 검증

### 첫 실행

1. 앱 데이터 경로와 로컬 라이브러리 생성
2. 교회 로그인 및 승인된 디바이스 등록
3. 기존 운영 폴더가 있으면 읽기 전용 검사 후 데이터 이관 제안
4. 성경·찬송가 라이선스 자료는 교회가 직접 선택해 설치
5. ATEM IP, Fill/Key 입력, Sub/Control 디스플레이 매핑
6. 테스트 패턴과 Linear Key 점검
7. 현장 프로필 저장 후 운영 화면 실행

업데이트 실패 시 새 앱만 롤백하고 로컬 라이브러리는 건드리지 않는다.

## 8. 구현 순서

1. 현재 ATEM 운영판 체크포인트 커밋
2. `unoworship-pro/apps/atem-field`에 코드만 클린 복제
3. 웹 빌드와 Electron 앱 빌드 경계 분리
4. 교회·회원·구독·장치 멀티테넌트 스키마 확정
5. 기존 헵시바 데이터에 교회 소유권 이관
6. 설교대지·준비찬양 입력 유형 추가
7. Electron 교회별 목록 조회와 수동 가져오기 구현
8. 오프라인 저장·재시도·중복 가져오기 방지 검증
9. macOS 서명·공증·DMG 현장 테스트
10. 수동 업데이트 배포

## 9. 정식 서비스 전 필수 통과 조건

- 다른 교회 자료가 검색·URL 변경·API 직접 호출로 노출되지 않는다.
- 구독 만료 교회는 새 클라우드 자료 사용이 차단되며 로컬 예배 안전 정책은 별도로 적용된다.
- 인터넷 단절 시 이미 가져온 프로그램을 정상 송출한다.
- 앱 업데이트 후 성경·찬송가·PPT·프로그램·녹화 파일이 유지된다.
- 클라우드 가져오기 실패가 기존 프로그램 파일을 훼손하지 않는다.
- 서비스 역할 키와 저작권 원본 데이터가 설치파일과 GitHub에 포함되지 않는다.

## 10. 2026-07-22 자체 점검 결과

| 점검 항목 | 현재 판단 | 필요한 개선 |
|---|---|---|
| GitHub 업로드 범위 | 대용량·저작권 데이터가 기존 폴더에 혼재 | `apps/atem-field` 클린 선별 복제와 비밀정보 검사 |
