## 1. 한 줄 정의

**담당자는 콘텐츠(제목·가사·본문)만 입력하면, 버튼 한 번으로 회중·무대 두 화면에
각자 맞는 디자인이 동시에 나가는 예배 방송 시스템.**

## 2. 왜 제품화하는가

현행 시스템(UnoLive-plus-atem-field)은 기능적으로 완성 단계에 도달했다.
- 듀얼아웃(출력별 개별 자막 라우팅) 실장비 검증 완료.
- 부팅 자동실행 → 3화면 배치 → 무인 운영 파이프라인 안정화.
- 개발용 멀티뷰(`/atem-dev`)로 ATEM 없이 개발 가능한 워크플로 확립.

그러나 제품이 되기엔 구조적 부채가 있다 (감사에서 확인).
- 에디터가 유기적으로 자라 송출 로직이 이중 구현되고, 스토어 전면 구독·핫패스 persist 등
  성능 부채가 iPad 안정성을 위협한다.
- dev 서버 운영이 강제되는 구조 (인증 우회가 NODE_ENV에 묶임) — 프로덕션 빌드 불가.
- UI가 기능 추가 순서대로 쌓여 신규 사용자(다른 교회 미디어팀)가 배우기 어렵다.

**결론 — 런타임(렌더러·소켓·듀얼아웃)은 계승하고, 에디터와 운영 구조는 새 디자인으로 재구축한다.**

## 3. 대상 사용자

| 사용자 | 상황 | 핵심 요구 |
|---|---|---|
| **예배 운영자** (주 사용자) | 예배 중 iPad로 원격 조작. 방송 지식 없음 | 실수해도 방송 사고가 안 나는 조작, 끊겨도 저절로 복구 |
| **콘텐츠 담당자** (성가대·설교·찬양팀) | 주중에 가사·본문 입력 | 디자인 몰라도 되는 템플릿, PPT/성경/찬송 가져오기 |
| **시스템 관리자** (미디어팀) | 초기 설치, 장비 연결 | 부팅만 하면 되는 무인 운영, 문제 시 진단 수단 |
| **리모트 사용자** (강대상 목사님·회중석 운영자) | 예배 중 iPad/모바일로 긴급 조작 | 큰 버튼 몇 개뿐인 특화 화면 — 실수 여지가 없는 최소 조작 |

## 4. 핵심 시나리오 (듀얼아웃 — 계승 기능)

한 번 송출 = 메인(회중, ATEM Fill+Key 선형 키)·서브(무대, **맥 직결 모니터**) 동시 도달.
각 출력이 자기 디자인만 렌더한다. 하드웨어 경로는 §5 참조.

| 시나리오 | 메인(회중) | 서브(무대) | 상태 |
|---|---|---|---|
| 성가 | 카메라 + 얇은 가사 | 검정 + 큰 흰 가사 + 다음 첫 줄 | 검증 완료 — 그대로 계승 |
| 텍스트/기타 | 개별 디자인 A | 개별 디자인 B (요소별 visibleOn) | 검증 완료 — 그대로 계승 |
| 설교 | 이미지/카메라 | 성경 본문 큰 글자 | 신규 구현 (레퍼런스에 스텁) |
| 영상 개별 송출 | 유튜브·업로드 영상 (필앤키 영상 — 검증 완료) | 별도 영상 or 무영향 (직결 로컬 재생) | 메인 검증 완료 · 서브는 직결 재생으로 신규 |
| 율동/찬양 | 유튜브·업로드 영상 | 안무 영상 + 가사 | 신규 구현 (부하 검토 필요) |

## 5. 자막 시스템 구조 (확정 2026-07-08)

### 5-1. 신·구 구조 비교

| | 구 구조 (레퍼런스) | **신 구조 (확정)** |
|---|---|---|
| 메인(회중) | 맥 C타입1/2 → ATEM 입력4(Fill)/5(Key) → Out1 | **동일 — 필앤키 유지** (카메라+자막 합성은 하드웨어 키잉이 정답) |
| 서브(무대) | 맥 HDMI → ATEM 입력6 → Out2 | **맥미니 직결** (ATEM 경유 제거) |
| 제어 | 없음 (컴포저는 iPad/노트북 IP 원격) | **맥미니 직결 모니터에서 로컬 조작 (기본)** |
| 리모트 | `http://맥IP:3000/composer` (전체 UI) | **긴급 조작 특화 레이아웃** (강대상·회중석용, 신규) |
| ATEM 입력6 | 서브 전용 소모 | **해방 → 카메라 입력 +1 여유** |

### 5-2. 물리 배선 (총 4화면)

```
맥미니
├─ C타입1 ──→ ATEM 입력4 (Fill) ─┐
├─ C타입2 ──→ ATEM 입력5 (Key) ─┼─ 리니어 키 → Out1 → 메인(회중) 모니터
├─ HDMI  ──→ 서브(무대) 모니터 직결   [권장]
└─ DisplayLink(USB) ──→ 제어 모니터    [권장]
```

**할당 권장안 — 서브=네이티브 HDMI, 제어=DisplayLink.**
DisplayLink는 USB 압축 그래픽이라 영상 재생 시 프레임 드랍 가능성이 있다.
서브는 영상 재생 대상이므로 네이티브 출력이 안전하고, 제어 화면(정적 UI)은 압축 영향이 없다.
대안 배치(HDMI=제어, DisplayLink=서브)도 가능하나 서브 영상 품질 검증 후에만.
최종 선택은 현장 리허설로 확정한다.

### 5-3. 출력 정의 (공식 명칭)

| 이름 | 화면 | 경로 | 내용 |
|---|---|---|---|
| **메인(main)** | 회중 모니터 | 필앤키 → ATEM Out1 | PGM 중심 (카메라+자막 합성). 섹션 디자인 그대로 |
| **서브(sub)** | 무대 모니터 | 맥 직결 | 검정배경 기본 + 조건부 다른 디자인·추가 자막 (듀얼아웃 계승) |
| **제어** | 제어 모니터 | 맥 직결 | 컴포저 로컬 조작 (기본) |
| **리모트** | iPad/모바일 | IP 접속 | 긴급 조작 전용 레이아웃 (송출/다음/이전/블랙아웃) |

### 5-4. 영상 송출 방식 (ProPresenter 수준 지향)

- 유튜브 링크·업로드 영상을 메인/서브에 **개별로** 띄울 수 있다 (같은 영상도, 서로 다른 영상도).
- **메인** — 필앤키 영상 방식 (Fill 재생 + Key 전체 마스크). 레퍼런스에서 실장비 검증 완료
  (`docs/features/video-fillkey/BRIEF.md`).
- **서브** — 직결이므로 합성 불요, 출력창 로컬 재생.
- 원칙 재정의: **카메라 믹싱·스위칭은 ATEM, 콘텐츠 영상(유튜브·업로드) 재생은 앱 담당.**

### 5-5. 출력 프로필 — 한 코드베이스, 두 구성 (확정 2026-07-08)

별도 폴더(unoworship-plus)로 포크하지 않는다. **하나의 앱에 출력 프로필 2종**을 두고
설정·기동 스크립트에서 선택한다. 기능은 100% 공통 — 수정 1번으로 두 구성에 반영.

| 프로필 | 배선 | 서브 화면 | 대상 |
|---|---|---|---|
| **A. 직결 (Pro)** | fill/key→ATEM 입력4/5→Out1(메인), 서브 모니터 맥 직결, 제어 DisplayLink | **개별 디자인** (검정배경 큰 가사 등 듀얼아웃 전체) | 기본 구성 (§5-2) |
| **B. 필앤키 (Plus)** | fill/key→ATEM 입력4/5, Out1→메인·Out2→서브, 제어 맥 HDMI | **PGM 미러** (메인과 같은 합성 화면) | ATEM 출력 2개를 쓰는 단순 구성 |

- 프로필 차이는 창 배치(기동 스크립트)와 서브 출력 경로뿐 — 컴포저·렌더러·소켓은 동일.
- Plus는 서브 개별 디자인이 없는 대신 배선이 단순하다 (직결 케이블 런 불필요).
- **프로필이 기능 노출을 결정한다** — Plus에서는 서브 개별 제어 계열 UI가 나타나지 않는다:
  서브 레이아웃(PMT) 선택, visibleOn 서브 라우팅, 섹션 타깃(메인만/서브만/같이),
  서브 전용 송출, 서브 개별 영상, 멀티뷰 SUB 창(→ "PGM 미러" 표시로 대체).
- 제품 티어 명칭으로도 사용: Pro(듀얼아웃) / Plus(미러).

**장비별 권장 매핑 (가이드 — 하드웨어 강제 아님)**

| 교회 장비 | 권장 프로필 | 근거 |
|---|---|---|
| ATEM 상위 기종(Extreme 초과) 또는 모니터 4대 여유 | **Pro (직결)** | 설치 규모가 되면 서브 개별 디자인의 가치가 큼 |
| ATEM Extreme (HDMI 출력 2) | **Plus** | Out1 메인 + Out2 서브 미러가 바로 됨 |
| ATEM Mini/Pro (HDMI 출력 1) | **Plus + HDMI 분배기** | 서브가 PGM 미러라 분배기로 동일 구성 가능 |

- Pro는 서브가 ATEM을 거치지 않으므로 기종 제한이 없다 — 기준은 모니터 수·배선 여건.
  Plus로 시작한 교회가 배선만 추가하면 프로필 전환으로 Pro 업그레이드.
- **원리: 개별 제어 = 별도 신호 공급.** ATEM 출력 개수는 조건이 아니다 — 출력(AUX)은
  기존 소스(PGM/입력/멀티뷰)를 고르는 라우팅일 뿐, 개별 자막 디자인은 맥이 만들어
  입력으로 넣어줘야 한다. 별도 신호 공급 방법은 두 가지: ①맥 출력을 ATEM 입력에 추가
  (구 구조 3입력 — 제어 모니터 포기) ②서브 맥 직결(Pro — DisplayLink로 제어 확보).

**프로필 C (후보, v2 검토) — 소프트웨어 합성 (vMix식)**

ATEM 영상을 맥으로 캡처해 앱 안에서 자막과 합성, 완성 화면을 모니터 창으로 직접 출력.
과거 버전에서 실동작 성공 이력 있음 (사용자 경험).

- 가치: **ATEM 없는 교회까지 커버** (USB 캡처만 필요) — 가장 넓은 시장. ProPresenter 지향과 부합.
- 기존 자산: ProgramMirror UVC 캡처 + 리니어 키 합성 알고리즘(검증 완료) — 프리뷰 품질을
  송출 품질로 올리는 과제로 좁혀짐.
- 리스크 (v1 제외 사유): 상시 실시간 합성은 정적 프레임 원칙의 상시 예외 — 과거 스터터
  사고의 경로. 2시간 무인 운영 부하 검증 + 캡처 지연(~100·200ms) 평가 후 v2에서 판단.

### 5-6. 현장 검증 필요 항목 (Phase 6 리허설에서 확정)

- DisplayLink 영상 재생 부하 — 프레임 드랍 여부 (권장 배치대로면 제어 화면이라 영향 최소).
- 직결 서브 창의 kiosk 자동 배치 (디스플레이 식별 — DisplayLink는 EDID가 고유해 배정 용이 예상).
- 필앤키(C타입 2개) 좌표순 구분은 구 구조와 동일 — 뒤바뀌면 ATEM 입력4·5 케이블 교환.

**사전 체험 실험 (DisplayLink 도착 전 — 현 장비 그대로 신 구조 체험 가능)**

1. 맥미니 HDMI를 ATEM 입력6에서 분리 → 무대 모니터에 직결 (`/atem-sub` 창이 그대로 표시됨).
2. 조작은 기존 IP 컴포저 (`http://맥IP:3000/composer`).
3. 필앤키(입력4/5) 방송 경로는 무영향 — 안전.

주의.
- ① 핫스왑 금지 — 모니터 EDID가 ATEM과 달라 디스플레이가 재배열될 수 있다.
  케이블 교체 후 3화면 실행 스크립트 재실행(또는 재부팅).
- ② 무대까지 배선 확인 — 기존 ATEM Out2 런이 HDMI인지·거리 (길면 액티브 HDMI 필요).
- ③ 실험 종료 복원 시에도 같은 재실행 절차 — 주중 실험, 예배 전 복원 확인.

확인 목표: 직결 화질/지연 체감, kiosk 창 배치 안정성, 운영 감각.

## 6. 새 UI 디자인 (Figma 프레임 1920x1080 기준)

제공된 디자인의 영역 해석. **⚠ 아래 매핑은 프레임 구조에서 추론한 가정이다 —
Phase 1 착수 전 사용자 확인 필수.**

```
┌────────────────────────────────────────────────────────────┐
│ ① 상단 바 (전폭)                                            │
├──────────┬────────────────────────────────────┬────────────┤
│ ② 좌측    │ ③ 중앙 대형 캔버스                   │ ⑤ 우측     │
│   사이드바 │    (편집 + PGM 프리뷰)               │   패널     │
│          ├────────────────────────────────────┤            │
│          │ ④ 하단 3분할                        │            │
│          │    (FILL / KEY / SUB 멀티뷰)         │            │
└──────────┴────────────────────────────────────┴────────────┘
```

| 영역 | 해석 (가정) | 근거·계승 요소 |
|---|---|---|
| ① 상단 바 | 프로그램 선택, 연결 상태 배지(출력·서버), LIVE 표시, 모드 전환(편집/운영) | 현행 MiddleTopMenu + 연결 배지 |
| ② 좌측 사이드바 | 세트리스트 — 상단 박스는 현재 프로그램 카드, 아래로 섹션 목록, 하단 박스는 가져오기(성경/찬송/PPT) 진입 | 현행 SetlistPanel |
| ③ 중앙 대형 | 편집 캔버스 ↔ PGM 합성 프리뷰(리니어 키 시뮬레이션) 토글. 운영 모드에선 PGM 고정 | 현행 EditorCanvas + `/atem-dev`의 키 합성 미리보기 |
| ④ 하단 3분할 | **듀얼아웃 상시 멀티뷰** — FILL/KEY/SUB 실시간 미리보기 (개발용이던 `/atem-dev`를 제품 기능으로 승격). SUB는 직결 모니터 창의 미리보기 (§5 신 구조) | `/atem-dev` OutputPane (실해상도 렌더 후 축소) |
| ⑤ 우측 패널 | 오퍼레이터 — 송출/다음/이전, 블랙아웃, 섹션 타깃(메인만/서브만/같이), promptLayout 선택 | 현행 OperatorPanel + SectionCueMacro |

디자인 원칙.
- **운영 모드와 편집 모드를 명시적으로 분리** — 감사에서 확인된 "편집 행위가 방송을 건드리는" 사고
  경로(마운트 시 송출, 편집 틱마다 소켓 전송)를 UI 레벨에서 구조적으로 차단.
- 뷰포트 반응형(32인치 기준, 스크롤 없음) — `/atem-dev`에서 검증한 셀 맞춤 스케일 방식.
- 모든 상태(LIVE 위치, 블랙아웃, 연결)는 항상 화면에 보인다 — 리로드 후에도 복원.

## 7. 기능 범위

### v1.0 (제품 최소선 — 현행 동등 + 안정화)
- 듀얼아웃 전체 (visibleOn 라우팅, promptLayout, 한 번 송출·양쪽 도달) — §5 신 구조(서브 직결) 기준
- 새 UI (6장 레이아웃) — 세트리스트, 편집 캔버스, 오퍼레이터, 상시 멀티뷰
- 리모트 오퍼레이터 — 긴급 조작 특화 레이아웃 (송출/다음/이전/블랙아웃만, 큰 버튼)
- 영상 개별 송출 — 유튜브·업로드 영상을 메인(필앤키 영상)/서브(직결 재생)에 독립 송출
- 리니어 키 합성 PGM 프리뷰 (카메라x(1-키)+필x키 — 검증된 알고리즘 이식)
- 가져오기 — 성경/찬송/PPT/유튜브 (현행 임포터 이식)
- 끊김 자동 복구 — 재연결 룸 재조인 + 서버 상태 리플레이 (감사 1순위 과제를 v1 기본값으로)
- iPad 리로드 완전 복구 — 라이브 위치·블랙아웃·편집본 보존
- 프로덕션 빌드 운영 + 부팅 자동실행 + 전체종료 (현행 스크립트 계승·개선)
- 진단 — 출력창 `?debug=1` 오버레이, 서버 상태 페이지

### v1.1+
- 설교 `bible` / 율동 `youtube-dance` promptLayout
- 시나리오 템플릿 (모드별 메인+서브 프리셋, 담당자는 텍스트만 입력)
- 커스텀 서브 디자인(`prompt-*`)의 ATEM 서브 렌더 (레퍼런스의 알려진 갭)
- 다교회 배포 패키징 (설치 스크립트, 환경 마법사)

### 비범위 (하지 않는 것)
- **카메라** 믹싱/스위칭 — ATEM의 일 (카메라 전환·키잉은 하드웨어에 맡긴다. 검증된 원칙).
  단 **콘텐츠 영상**(유튜브·업로드) 재생은 앱 담당 (§5-4)
- 저작권 성경/찬송 텍스트 번들 — 교회 보유/허가 자료 직접 입력만
- 클라우드 스트리밍 송출 — LAN 로컬 운영이 1차 (오프라인 예배당이 기본 환경)

## 8. 성공 기준

1. **방송 사고 0** — 운영자 명시 조작 없이는 어떤 코드 경로도 송출 상태를 바꾸지 못한다.
2. **자동 복구** — WiFi 순단·서버 재시작·iPad 리로드 후 10초 내 무개입 원상복구.
3. **2시간 무인 운영** — 메모리·CPU 평탄 (정적 프레임 원칙 유지).
4. **신규 교회 온보딩 1일** — 설치부터 첫 예배 운영까지.
